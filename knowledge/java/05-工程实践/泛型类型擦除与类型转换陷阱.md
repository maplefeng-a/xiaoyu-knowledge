# 泛型类型擦除与类型转换陷阱

## 问题场景

在 `SkillToolHandlers.skillList()` 中，从 `Map<String, Object>` 取出 `List` 并强制转换时抛出异常：

```
class java.util.LinkedHashMap cannot be cast to class com.assistant.mcp.model.SkillSummary
```

## 根因分析

### 数据流

```
Service 层                              Handler 层
    ↓                                       ↓
List<SkillSummary> skills                 
    ↓
.map(SkillSummary::toMap).toList()       
    ↓                                      ↓
List<Map<String, Object>>  ──────→  (List<SkillSummary>) result.get("skills")
                                              ↓
                                         ClassCastException!
```

### 核心概念：泛型擦除

Java 泛型在**编译时**检查类型，在**运行时**被擦除：

```java
// 编译前
List<SkillSummary> list1 = new ArrayList<>();
List<Map<String, Object>> list2 = new ArrayList<>();

// 编译后（字节码层面）
List list1 = new ArrayList();  // 泛型参数被擦除
List list2 = new ArrayList();  // 两者变成相同的原始类型
```

### 为什么编译通过但运行时报错？

```java
// 这行代码编译通过
List<SkillSummary> skills = (List<SkillSummary>) result.get("skills");

// 运行时只检查了 "是否是 List"，没有检查泛型参数
// 因为泛型参数在运行时已经不存在了！
```

### 异常发生的真正位置

```java
for (SkillSummary skill : skills) {
    skill.name();  // ← 这里报错
}

// 编译器隐式插入的代码：
for (Object obj : skills) {
    SkillSummary skill = (SkillSummary) obj;  // ← 实际报错位置
    //                    ↑
    //            obj 实际是 LinkedHashMap，不是 SkillSummary
}
```

## 正确的处理方式

### 方式1：按实际类型处理（推荐）

```java
@SuppressWarnings("unchecked")
List<Map<String, Object>> skills = (List<Map<String, Object>>) result.get("skills");

for (Map<String, Object> skill : skills) {
    String name = (String) skill.get("name");
    String description = (String) skill.get("description");
    // ...
}
```

### 方式2：保持 Service 返回原始对象

```java
// Service 层：直接返回 List<SkillSummary>，不转换为 Map
public List<SkillSummary> listSkills() {
    // ...
    return skills;  // 不要 .map(SkillSummary::toMap).toList()
}
```

## 关键知识点

| 概念 | 说明 |
|------|------|
| **泛型擦除** | 运行时泛型参数被移除，`List<T>` 变成 `List` |
| **原始类型** | 泛型擦除后的类型，如 `List`、`Map` |
| **桥接方法** | 编译器为泛型继承生成的方法 |
| **堆污染** | 泛型类型不一致导致的运行时问题 |

## 经验法则

1. **从 `Map<String, Object>` 取出的值，按实际类型处理**
   - 不要假设泛型参数在运行时还存在

2. **强制转换 `List<T>` 只检查 List，不检查 T**
   - 运行时无法验证泛型参数

3. **使用 `@SuppressWarnings("unchecked")` 时确保逻辑正确**
   - 压制警告的前提是你知道自己在做什么

4. **优先使用具体类型而非 Object**
   - 能在编译时发现的问题，不要留到运行时
