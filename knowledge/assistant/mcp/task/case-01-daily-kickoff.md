
# Case 1：晨间主动开工（Daily Kickoff）

## 1. 目标
- 本 case 要解决的问题：
  - 每天开始工作时，用户不需要手动整理上下文，Agent 主动给出"今日可执行计划"。
- 成功标准（可验证）：
  - Agent 在触发条件满足时主动发起。
  - 用户通过自然语言确认后成功生成当日计划文件。
  - 计划包含 2-4 个锚点任务，结构符合既有 `tasks` 模板。

## 2. 触发条件
- 时间触发：
  - 工作日 09:30（可配置）。
- 事件触发：
  - Agent 进程启动后首次进入工作时段。
- 前置条件：
  - 当天 `tasks/YYYY-MM-DD.md` 不存在，或存在但"今日任务"为空。

## 3. 输入上下文
- 必读数据源：
  - `memory/anchors.md`
  - `tasks/backlog.md`
  - 最近 3-7 天 `tasks/*.md`（通过 `task-context` 获取索引）
  - 最近若干 `messages/` 进展记录（可选）
- 关键字段：
  - 昨日未完成事项
  - 时间敏感事项
  - 当前高优先级锚点

## 4. Agent 主动消息
- 首条消息模板：
  - "早上好，我已基于昨日进展和 backlog 生成今日计划草案（3 项重点）。是否采用？你也可以让我调整优先级。"
- 必须包含：
  - 建议清单（Top3）
  - 理由（为什么是这3项）
  - 影响（不做会影响什么）
  - 待确认动作（确认后写入今日计划）

## 5. 用户交互分支
- 分支A（确认）：
  - 用户自然语言回复"确认" -> 进入写入流程。
- 分支B（修改）：
  - 用户自然语言回复"第2项改成X" -> Agent 重生成草案并再次确认。
- 分支C（拒绝/延后）：
  - 用户回复"今天先不排" -> Agent 记录状态，不写文件，午间再次提醒。

## 6. MCP 工具调用流程 (v2.0.0)

**新流程**（2 个 MCP 工具）：

```
1. task-context(days=7, include_done=true)
   ↓ 获取近期任务索引
2. Read 工具（按需读取详情）
   ↓ Agent 自主决策
3. Agent 生成计划内容
   ↓ 无 MCP 调用
4. （等待用户确认）
5. task-write(date=today, mode=create, content=...)
   ↓ 落盘
```

**每步关键输出**：
- `task-context`：entries（日期 + 任务列表）
- `task-write`：success, written, conflict, file_path

**与旧版本差异**：

| 旧版本 | 新版本 (v2.0.0) |
|--------|----------------|
| `task_context_scan` | `task-context` |
| `task_daily_propose` | Agent 自己生成 |
| `task_daily_upsert` | `task-write` |

## 7. 写入与变更规则
- 写入文件：
  - `tasks/YYYY-MM-DD.md`
- 文件格式：
  ```markdown
  ---
  date: YYYY-MM-DD
  tasks:
    - 任务1
    - 任务2
  ---

  # 今日计划

  日期：YYYY-MM-DD

  ## 重点
  - 核心任务1

  ## 任务
  - [ ] 任务1
  - [ ] 任务2
  ```
- 允许修改范围：
  - 首次仅创建"今日任务"与基础头部；不覆盖已有"今日总结"。
- 幂等/覆盖策略：
  - `mode=create`：已存在且非空时返回冲突，不覆盖
  - `mode=overwrite`：强制覆盖（需用户明确确认）
- 审计字段：
  - `request_id`

## 8. 异常与回退
- 常见错误：
  - 文件权限不足
  - 路径不存在
  - 上下文读取失败
  - 写入冲突
- 降级策略：
  - 降级为"仅展示草案，不写入"
- 用户提示：
  - "我已生成草案，但写入失败。是否稍后重试？"
  - "今日计划已存在，是否需要更新？"

## 9. 验收标准
- 功能验收：
  - 触发后能主动提案，并在确认后落盘。
- 数据验收：
  - 文件结构符合现有 `tasks` 格式，含 YAML frontmatter 和 checklist。
- 交互验收：
  - 至少一次明确确认后才发生写入。
  - 写入后用户可通过 `/plan` 查看结果。

## 10. CLI 交互约束（更新）
- 主入口：Agent 主动消息 + 用户自然语言确认/修改。
- `/plan`：仅用于查看今日计划，不承担创建动作。

## 11. 对应开发 Feature 映射

| 组件 | 职责 |
|------|------|
| **assistant-mcp** | `task-context`（索引获取）、`task-write`（标准化写入） |
| **assistant-agent** | 读取详情（Read 工具）、生成计划内容、确认状态机 |
| **assistant-cli** | 主动提醒展示、自然语言确认/修改输入、`/plan` 只读查看 |
| **assistant-eval** | 用例：触发 -> 提案 -> 确认 -> 文件生成 -> `/plan` 查看一致性校验 |
