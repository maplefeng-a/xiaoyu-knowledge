# assistant-desktop 设计语言规范与交互检查（v0.1）

## 背景

本文档用于沉淀 `assistant-desktop` 当前阶段的系统级交互设计共识，避免后续优化陷入局部样式调整。

定位：
- 左侧以对话为主（生产区）
- 右侧以查看为主（工作区）
- 优化重点是信息层级、状态语义与一致性，而不是单点视觉花样

---

## 1. 设计语言规范草案（v0.1）

### 1.1 系统定位

- 产品定位：对话驱动的个人工作台
- 主轴关系：左侧生产（对话），右侧查看（工作区）
- 设计原则：内容优先、状态明确、工具克制

### 1.2 容器层级规范

- `L1 主容器`：页面级区域（聊天区、workspace 区）
  - 视觉特征：低对比背景、大块分区、边界清晰
- `L2 功能容器`：每个 tab 的主面板（知识树、消息列表、技能详情）
  - 视觉特征：统一圆角/边框/背景，强调内容承载
- `L3 工具条/状态条`：搜索、刷新、统计、过滤
  - 视觉特征：更轻、更薄，不抢内容注意力
- `L4 内容单元`：树节点、消息项、技能项、卡片项
  - 视觉特征：可点击态明确，默认低噪音

### 1.3 状态规范（统一）

- `loading`
  - 结构：标题 + 简短状态文案
  - 禁止：大块空白无反馈
- `empty`
  - 结构：当前状态说明 + 下一步动作提示
  - 原则：不占过高空间
- `error`
  - 结构：错误信息 + 恢复动作（刷新/重试）
  - 原则：错误尽量局部展示，不吞掉已有内容
- `ready`
  - 原则：减少提示，直接展示内容

### 1.4 控件规范

- `搜索框`
  - 同一模块只保留一个主搜索入口
  - 宽度应服务模块任务（例如知识树搜索应明显宽于 badge）
- `次级按钮（刷新等）`
  - 统一尺寸、低视觉权重
  - 靠近对应内容区，不做全局主按钮
- `badge`
  - 仅做状态补充，不应压过主标题文本
- `tab`
  - 只负责切换，不承担复杂状态说明

### 1.5 信息层级规范（重点）

- 每个模块同一时刻只允许一个主焦点
  - 知识模块：首页看树，阅读模式看正文
  - 消息模块：消息列表或当前正文
- 同一区域避免“双主工具条”
  - 工具条与统计条必须主次分明

### 1.6 Markdown 阅读规范（系统资产）

- 右侧 Markdown 区按“阅读器”标准维护，而不是普通面板
- 保持稳定的：
  - 标题层级
  - 段落间距
  - 表格可读性
  - 代码块对比度
  - 长文滚动舒适度

### 1.7 联动规范（当前阶段）

- 仅做 `弱联动提示`
- 不自动切 tab、不抢焦点
- 联动优先由用户显式触发（点击）

## 2. 基于规范的当前实现检查（现状评估）

### 2.1 系统主轴（左生产 / 右查看）

结论：`基本达标`

优点：
- 左侧对话区已形成主任务路径
- 右侧工作区承担查看与辅助操作，结构成立

风险点：
- 某些时刻右侧内容视觉重量偏高，会与聊天区抢焦点（P2）

### 2.2 知识模块（当前重点）

结论：`方向正确，仍需层级收敛`

已做对的：
- 阅读模式从树模式解耦（首页选 / 阅读模式看）
- 去掉低价值“暂无文档内容”占位容器
- 默认展开层级优化（当前为 2 级，空目录不自动展开）
- 顶部已收敛为“搜索为主、树头刷新”为辅

仍可优化：
- 树头标题（如“知识树”）权重可进一步弱化，避免与顶部搜索竞争
- 目录节点/文档节点/badge 的视觉权重仍可再拉开（P2）

### 2.3 状态规范一致性（全系统）

结论：`部分达标`

已做对的：
- 明显减少无意义空态占位
- 阅读模式加载/错误状态可见

问题：
- 不同模块的 loading/empty/error 结构与文案风格尚未完全统一（P1）
- 部分错误文案偏技术化，恢复动作与错误提示距离较远（P2）

### 2.4 控件规范（搜索 / 刷新 / tab）

结论：`中等达标`

已做对的：
- 知识模块主搜索入口已上移并独立
- 刷新按钮职责明确，留在树容器头部

问题：
- 搜索框尺寸与布局曾出现反复，说明缺少模块级控件尺寸规范（P1）
- 各模块次级按钮视觉权重可能不完全一致（P2）

### 2.5 Markdown 阅读体验

结论：`当前可接受（按使用反馈）`

- 当前不是主要矛盾
- 建议后续以“阅读规范”方式固化，避免被其它样式改动影响

### 2.6 跨区联动

结论：`当前不做强联动是正确策略`

原因：
- 智能体动作具有不确定性
- 强联动（自动切 tab / 自动跳转）易误触发、打断用户

建议：
- 后续优先做“弱联动提示”（提示点、引用提示、用户点击后再联动）

---

## 3. 后续优先级建议（不涉及大功能）

### P1（优先）

1. 统一状态规范（loading / empty / error / ready）
2. 建立控件尺寸规范（尤其搜索框、次级按钮）
3. 收敛知识模块顶部层级（搜索条 vs 树头标题主次）

### P2（次级）

1. 调整知识树节点视觉权重（目录 / 文档 / badge）
2. 弱联动提示规范（先定义，不急实现）
3. 右侧工作区与左侧聊天区的视觉主次微调

---

## 4. 当前共识（阶段性）

- 不继续强行推进图谱首页
- 知识模块以目录树作为主结构入口是可接受的
- 优化重点从“样式尝试”转向“规范化与一致性”
- 先把系统交互语言定稳，再做增量功能（如学习闭环、弱联动）
