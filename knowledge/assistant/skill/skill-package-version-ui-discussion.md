# Skill 包多版本管理 UI 讨论稿（assistant-desktop）

## 目标（已收敛）

这不是运行态管理面板，也不是安装/导入管理面板。

首版定位为：`技能包信息浏览 + 多版本管理展示`

核心目标：

1. 展示技能包列表（按 skill 聚合）
2. 展示每个 skill 的多版本信息
3. 查看某个版本的元信息（名称、版本、描述、文件名等）
4. 为后续版本管理动作预留位置（动作本身待接口确认）

## 非目标（当前不做）

- 运行态状态（启用/禁用/异常）
- 安装/卸载/导入流程
- 仓库 vs 本地来源区分
- 任务进度/日志/异常面板

## 放置位置（贴合现有 desktop 结构）

基于当前 `assistant-desktop` 的三段式布局：

- 左：`Sidebar`
- 中：`ChatWindow`
- 右：`WorkspacePanel`

建议在 `WorkspacePanel` 中新增 Tab：`技能包`

## 多版本管理逻辑（UI 侧）

### 1. 列表按 skill 聚合

- 列表一行代表一个 `skill`
- 不按每个版本单独占一行（避免右侧面板过载）

列表最小展示建议：

- `name`
- `latest_version`
- `version_count`
- `description`（单行）

### 2. 详情区展示版本列表

- 点击列表项或 `详情` 后展开详情区
- 默认选中最新版本
- 在详情区展示该 skill 的 `versions[]`

版本列表展示可先用简单列表（首版不做复杂控件）：

- `1.0.3`
- `1.0.1`
- `1.0.0`

### 3. 当前选中版本信息

详情区中，版本列表下方展示“当前选中版本”的元信息。

建议最小字段：

- `version`
- `file_name`（如有）
- `description`（如有）
- 其他元信息（按后端实际返回决定）

### 4. 版本管理动作区（预留，不预设）

- 首版可为空
- 若后端后续确认支持版本管理动作，再按能力显示按钮
- 当前不预设动作集合，避免前端设计先行绑定实现

## 与 mcp-dev 的讨论清单（未定稿）

当前只确认“需要的能力”，不确认接口路径/字段定稿。

需要讨论确认：

1. 数据返回模型
- 后端是否按 `skill` 聚合返回（推荐）
- 还是按文件列表返回，由前端聚合（不推荐）

2. 多版本数据位置
- 列表接口直接带 `versions[]`
- 还是详情接口按需返回 `versions[]`

3. 版本元信息最小字段
- 至少需要哪些字段（版本号、文件名、描述、时间等）

4. 首版是否支持版本管理动作
- 若支持，动作集合是什么（例如“标记默认版本/删除版本”等，待后端确认）
- 若不支持，则首版仅做浏览与查看

## 给 mcp-dev 的一句话对齐结论

前端首版先做“技能包多版本浏览”，重点是 `按 skill 聚合 + 版本列表展示`；版本动作和接口细节等你们确认能力后再补。
