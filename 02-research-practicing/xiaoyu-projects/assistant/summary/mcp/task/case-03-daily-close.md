
# Case 3：收工主动复盘（Daily Close, 23:00）

## 1. 目标
- 本 case 要解决的问题：
  - 收工时自动完成“今日复盘沉淀”，减少遗忘与重复整理成本。
- 成功标准（可验证）：
  - 到达收工触发条件后，Agent 主动给出复盘草案。
  - 用户确认后，`今日总结` 被正确回写到当日任务文档。
  - 草案包含完成情况、问题、次日建议。

## 2. 触发条件
- 时间触发：
  - 工作日 23:00（晚上11点，可配置）。
- 事件触发：
  - 用户自然语言输入“收工”“今天结束了”“做下总结”。
- 前置条件：
  - 当日 `tasks/YYYY-MM-DD.md` 存在。
  - 若总结已存在完整内容，则走“更新总结”模式。

## 3. 输入上下文
- 必读数据源：
  - 当日 `tasks/YYYY-MM-DD.md`
  - 当日相关 `messages/` 记录（可选，用于补充事实）
- 关键字段：
  - checklist 完成状态
  - “完成情况”文本
  - 当日新增/调整记录（如有）

## 4. Agent 主动消息
- 首条消息模板：
  - “已到 23:00，我已基于今日进度生成复盘草案：完成项、未完成项、主要问题和明日建议。请确认或补充后我再写入。”
- 必须包含：
  - 完成项摘要
  - 未完成项与原因
  - 问题与阻塞
  - 明日 Top3 建议
  - 待确认动作（确认写入）

## 5. 用户交互分支
- 分支A（直接确认）：
  - 用户：“确认发布” -> 回写总结。
- 分支B（补充修改）：
  - 用户：“问题里补充 X，明日建议改成 Y” -> 重新草案 -> 再确认 -> 写入。
- 分支C（仅保存草稿）：
  - 用户：“先存草稿，稍后再发” -> 仅会话保存，不写文件。
- 分支D（跳过）：
  - 用户：“今天不总结” -> 记录 skipped，次日晨间提示补录。

## 6. MCP 工具调用流程 (v2.0.0)

**新流程**（2 个 MCP 工具）：

```
1. task-context(days=1, include_done=true)
   ↓ 获取今日任务索引
2. Read 工具读取 tasks/YYYY-MM-DD.md
   ↓ Agent 理解当前计划完成情况
3. Agent 生成复盘总结
   ↓ 无 MCP 调用
4. （等待用户确认）
5. task-write(date=today, mode=overwrite, content=...)
   ↓ 覆盖写入（含今日总结）
```

- 每步关键输出：
  - `task-context`：今日任务列表
  - `Read`：完整计划内容
  - `task-write`：success, written, file_path

## 7. 写入与变更规则
- 写入文件：
  - `tasks/YYYY-MM-DD.md`
- 允许修改范围：
  - `## 今日总结` 区块
  - 必要时补齐任务“完成情况”
- 幂等/覆盖策略：
  - 默认覆盖当日总结区；保留历史版本可由 Git 追踪
- 审计字段：
  - `requestId`
  - `change_reason = daily_close_summary`

## 8. 异常与回退
- 常见错误：
  - 总结区块解析失败
  - 文件不存在
- 降级策略：
  - 不写文件，返回可复制草案文本
- 用户提示：
  - “总结草案已生成，但写入失败。是否让我重试？”

## 9. 验收标准
- 功能验收：
  - 支持主动复盘与用户触发复盘两条入口。
- 数据验收：
  - `今日总结` 字段完整且结构稳定（收获/问题/建议）。
- 交互验收：
  - 确认前不写入；修改后可二次确认再写入。
  - 写入后用户可通过 `/plan` 查看当日完整状态。

## 10. CLI 交互约束（更新）
- 主入口：23:00 Agent 主动发起，或用户自然语言触发复盘。
- `/plan`：仅用于查看当日计划与总结，不承担总结写入入口。

## 11. 对应开发 Feature 映射
- assistant-mcp：
  - `task_daily_get`
  - `task_daily_propose(mode=close_summary)`
  - `task_daily_summary`
- assistant-agent：
  - `daily-planning-skill`（close 子流程）
  - 总结草案生成与确认状态管理
- assistant-cli：
  - 23:00 收工提醒展示
  - 草案编辑与确认
  - `/plan` 只读查看
- assistant-eval：
  - 用例：23:00 触发 -> 草案生成 -> 确认回写 -> `/plan` 查看一致性校验
